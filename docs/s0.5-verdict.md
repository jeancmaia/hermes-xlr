# S0.5 Bring-up Spike — Go/No-Go Verdict

**Date:** 2026-06-22
**Engine:** llama-server v1 (e365e65) via Vulkan
**Model:** Nemotron-Mini-4B-Instruct Q4_K_M GGUF
**GPU:** NVIDIA GeForce RTX 3050 6GB Laptop GPU (CUDA 572.40)

## ⚡ Verdict: **GO**

The capacity hypothesis is confirmed. A 6 GB GPU can serve a ~4B INT4 model
for agent use with comfortable headroom and working prefix cache.

---

## Measured numbers

| Metric | Value | Notes |
|---|---|---|
| **VRAM (idle)** | 6002 MiB free | Available before load |
| **VRAM (loaded)** | 3178 MiB used | 33/33 layers GPU-resident |
| **VRAM headroom** | ~2825 MiB | ~2.8 GiB free during inference |
| **Decode througput** | ~45 tok/s | 22 ms/tok, stable |
| **Prefill througput** | 75–105 tok/s | 202-token prompt |
| **TTFT (short prompt)** | 288 ms | 26 tokens, no cache |
| **TTFT (cached prefix)** | 61 ms | 26/45 cached, 4.7× speedup |
| **Peak VRAM observed** | 3179 MiB | No growth during decode |

## Key findings

1. **VRAM budget fits comfortably.** The model uses ~3.2 GB, leaving ~2.8 GB
   for OS, agent process, and headroom. This aligns with the HER-3 budget estimate
   of ~3224 MB total.

2. **Decode speed is adequate.** ~45 tok/s is fast enough for interactive agent
   use (a 200-token response in ~4.4 s). For contrast, human reading speed is
   ~5–10 tok/s.

3. **Prefix caching works.** The slot-based KV cache gives a 4.7× TTFT speedup
   on follow-up turns — critical for multi-turn agent sessions.

4. **Vulkan backend is a limitation.** The shipped `llama-server` binary uses
   Vulkan (not CUDA). CUDA would likely improve tok/s and reduce VRAM overhead.
   The project should prefer a CUDA build or ensure the transport can select
   the backend.

## Risks & caveats

- **Single slot.** Only one concurrent slot was tested. Agent use with parallel
  tool calls or multiple slots will increase KV cache memory.
- **No tool calls tested.** The completion tested was plain text. Tool-call mode
  (structured JSON output) may add latency.
- **Vulkan vs CUDA overhead.** The Vulkan path adds memory and latency that a
  native CUDA build would not. This may matter at lower VRAM margins.

## Recommendation

Proceed to **S2 (Backend seam + native bring-up)** with confidence. The numbers
support the core thesis. Priorities for S2:

1. Use a **CUDA build** of llama.cpp for lower overhead.
2. Validate **tool-call structured output** latency.
3. Begin the **backend-abstracted seam** design.
